Programming Version

2024-03-06 | programming | Hits

軟體版本號怎麼訂？只要，團隊可以正確理解並遵循就好。

https://tom.preston-werner.com/2022/05/23/major-version-numbers-are-not-sacred

為何要有版本號？
因為軟體會不斷進化、迭代。

假設我們推出一個 API（或是套件、服務）讓客戶使用，如果沒有壓上版本號，客戶順利的使用了第一個版本後，我們在第二個版本可能調整了呼叫 API 的輸入、輸出介面或參數。我們通知客戶使用新版何時會上線，這時候客戶如果沒有在版本更新時將所有 request response 根據最新版本規格調整好，就會出現與新版本系統不相容的狀況。甚至即使客戶馬上將所有 request response 調整了，也可能因為舊版的一些 request 還沒跑完但系統已經更新而發生錯誤。

因為要兼容不同版本的系統使用，所以有了版本號這個東西。也就是說，系統的使用者或客戶在軟體更版的時候，得以辨識他們拿到的系統版本是哪一版，做對應的處理（可能分開處理升級與尚未升級的部分）。

作為一名專業的開發者、或是一間專業的軟體服務提供者，如果改變對你的使用者是重要的，就要透過版號真實傳達修訂的意義。

保持專案高效率的一個重要部分是掌控軟體複雜度，如果今天沒有人知道使用你的軟體哪些函數可以可靠的呼叫，就無法好好的控制複雜度。

事實上，生活處處皆版號，例如我現在打文章用的 macOS 是 Big Sur 版本是 11.2.1、我上班的 Windows 版本是 Windows 10、我的 Python 版本是 3.8.7、同事借我的計算機概論是第 4 版（這個的意義應該就跟軟硬體不大相同了，但提供辨識版本別的機制這主軸類似）……。

而相容性 (compatibility) 就是系統發生變更後表現一致的特性。

Semantic Versioning
這個版本制定格式中文翻譯為「語意化版本」，是由 Tom Preston-Werner 提出的版本號訂製規則。

A meaningful method for incrementing version numbers.

按照 Semantic Versioning 文件，版本格式定義如下：

MAJOR.MINOR.PATCH

也就是「主版號.次版號.修訂號」，例如：1.0.0。

而版號遞增規則如下：

主版號：當你做了不相容的 API 修改，
次版號：當你做了向下相容的功能性新增，
修訂號：當你做了向下相容的問題修正。
先行版號及版本編譯資訊可以加到「主版號.次版號.修訂號」的後面，作為延伸。

Backward compatibility is a property of a system, product, or technology that allows for interoperability with an older legacy system, or with input designed for such a system, especially in telecommunications and computing.

A complementary concept is forward compatibility. A design that is forward-compatible usually has a roadmap for compatibility with future standards and products.

向下相容 (Upward compatibility) = 向後相容 (Backward compatibility)：意指當系統（如程式、library 等）更新後，使用較落後版本建立的檔案、系統仍可以在新的系統中正常使用，或用舊版本仍可正常編譯執行

向上相容 (Downward compatibility) = 向前相容 (Forward compatibility)：意指舊版本系統（如程式、library 等）可以支援新版本的資料或服務
